Azure Cloud Security for Agentic AI

A security-architecture checklist for building agentic-AI workloads on Azure — 40 controls across eight domains, each mapped to SOC 2 Type II, CMMC L1, CMMC L2, HIPAA, EU CRA, GDPR, and Azure best practices. Every frame is collapsible; each item carries a description, an implementation, a continuous-verification method, and a check-off with notes. Filter by framework, priority, or the agentic-AI focus.

Terminology & how to use this

Each control is collapsible by domain, tagged to all seven frameworks, prioritized P1–P2, and has a check-off with a notes field (saved in this browser). The Agentic AI security domain is the focus of this checklist. Reference terms:

Agentic AI
AI systems where a model plans and takes actions through tools/APIs — not just answering, but doing. Security must constrain what the agent can do and prove what it did.
Amazon Azure AI Foundry
Azure managed service for foundation models and agents; provides model access, Agents, Knowledge Bases, and Guardrails.
Azure AI Content Safety
Configurable input/output filters for models: denied topics, PII/PHI redaction, and prompt-attack detection.
Azure AI Agent Service
Managed agents that call tools (“action groups”) and knowledge bases under a scoped execution role.
Return of control
A Azure AI Foundry pattern where the agent hands a proposed action back to your application (or a human) instead of executing it directly.
Prompt injection
An attack where untrusted content (a document, tool output, web page) smuggles instructions that hijack the agent. Treat all such content as untrusted.
RAG
Retrieval-Augmented Generation — grounding a model on retrieved documents (a vector store / knowledge base) rather than only its training.
MCP
Model Context Protocol — a standard for connecting agents to external tools/data servers; third-party MCP servers are supply-chain surface to vet.
Human-in-the-loop
A required human approval step before an agent performs a high-impact or regulated action.
Azure RBAC
Azure Identity and Access Management — users, roles, and policies that govern who/what can call which Azure APIs.
Microsoft Entra ID
Azure SSO service; federates human access to your identity provider with permission sets instead of standing standing user accounts.
Role / STS
An Azure RBAC identity assumed temporarily; STS issues short-lived credentials so nothing long-lived is stored.
Azure Policy
Azure Policy — an organization-wide guardrail that caps what any account/principal can do, regardless of their Azure RBAC policy.
Azure Key Vault / customer-managed key
Key Management Service; a customer-managed key (customer-managed key) encrypts data at rest with policies and rotation you control.
storage public-access disabled + private endpoints
Account/bucket setting that prevents Blob Storage data from being exposed publicly by policy or ACL.
VPC / network security group (NSG)
Virtual Private Cloud network isolation; a network security group (NSG) is a stateful virtual firewall on a resource’s network interface.
Azure Private Link / Private Endpoint
Private network paths to Azure or partner services so traffic never traverses the public internet.
Azure WAF / Azure DDoS Protection
Azure Web Application Firewall (managed rules, rate limiting) and Azure DDoS Protection (DDoS protection) at the edge.
Microsoft Defender for Cloud
Managed threat detection across accounts, workloads, and data planes (malware, AKS, Azure SQL, Azure Functions, Blob Storage).
Microsoft Defender for Cloud (CSPM)
Cloud security posture management — aggregates findings and scores config against benchmarks (Microsoft cloud security benchmark, CIS).
Azure Policy
Records resource configuration and evaluates it against rules/policy initiatives continuously.
Azure Monitor activity log
Records every Azure API call — the primary audit log; can be made immutable in a locked log-archive account.
Microsoft Purview
Managed sensitive-data discovery for PII/PHI in Blob Storage.
Defender vulnerability management
Continuous vulnerability scanning for EC2, containers, and Azure Functions.
Azure Key Vault
Stores and rotates secrets; applications fetch them at runtime by role instead of embedding them.
Azure Landing Zones
Sets up and governs a multi-account landing zone with guardrails.
Microsoft Purview Compliance Manager
Automates evidence collection mapped to frameworks (SOC 2, HIPAA, etc.).
SOC 2 Type II
An attestation over a period that controls meeting the Trust Services Criteria (TSC) operate effectively. CC = Common Criteria; A/C/PI/P = Availability, Confidentiality, Processing Integrity, Privacy.
CMMC Level 1
The basic-safeguarding tier (17 practices from FAR 52.204-21) protecting Federal Contract Information (FCI). Physical (PE) practices are largely inherited from Azure.
FCI
Federal Contract Information — non-public info provided/generated under a federal contract; the data CMMC L1 protects.
HIPAA Security / Privacy Rule
Security Rule = administrative (§164.308), physical (§164.310), technical (§164.312) safeguards for ePHI. Privacy Rule = use/disclosure limits, minimum necessary (§164.502), de-identification (§164.514).
BAA
Business Associate Agreement — required with Azure for services processing PHI.
EU CRA
EU Cyber Resilience Act — essential cybersecurity requirements for products with digital elements. Annex I Part I = product requirements; Part II = vulnerability handling (incl. coordinated disclosure).
CMMC Level 2
The advanced tier — all 110 NIST SP 800-171 practices — required to handle Controlled Unclassified Information (CUI). Where CMMC L1 stops, L2 adds encryption, deep audit, configuration and incident-response practices; controls that ‘exceed L1’ here are covered at L2.
GDPR
EU General Data Protection Regulation. This checklist maps the technical security-of-processing measures (Art. 32) and related articles (5, 25, 30, 33/34, 44); GDPR also requires non-technical measures — lawful basis, consent, data-subject rights (Art. 15–22), DPO, DPIAs — that sit outside a technical checklist.
SBOM
Software bill of materials — a component inventory; a CRA expectation for products with digital elements.
Azure Well-Architected Framework Security Pillar
Azure’s security best-practice framework (SEC01–SEC11): identity, detection, infrastructure/data protection, incident response.

Design decisions — agentic-AI specific

  • The agent is a principal, not a person — it gets its own least-privilege role, its own audit identity, and never a human's credentials. (access control across every framework)
  • All model, tool, and RAG content is untrusted input — Guardrails + output validation defend against prompt injection and data exfiltration. (SOC 2 PI · CRA integrity · HIPAA)
  • High-impact actions require a human — destructive, financial, or PHI actions pass through an explicit approval step. (HIPAA §164.312(a) · CMMC AC.L1-3.1.2)
  • Every action is provable — full trace logging (prompt, retrieval, tool call, output) to an immutable store is the audit and assessor evidence. (HIPAA §164.312(b) · SOC 2 CC7.2)
  • Blast radius is bounded by design — per-tool sandboxes, per-tenant data isolation, egress allow-lists, and rate/cost limits contain a compromised agent. (CRA attack-surface · availability)
  • Data stays where it must — region-pinned models, per-classification KBs, and minimum-necessary retrieval keep PHI/FCI inside the boundary. (HIPAA privacy · CMMC · CRA data minimization)
