Google Cloud Cloud Security for Agentic AI

A security-architecture checklist for building agentic-AI workloads on Google Cloud — 40 controls across eight domains, each mapped to SOC 2 Type II, CMMC L1, CMMC L2, HIPAA, EU CRA, GDPR, and GCP best practices. Every frame is collapsible; each item carries a description, an implementation, a continuous-verification method, and a check-off with notes. Filter by framework, priority, or the agentic-AI focus.

Terminology & how to use this

Each control is collapsible by domain, tagged to all seven frameworks, prioritized P1–P2, and has a check-off with a notes field (saved in this browser). The Agentic AI security domain is the focus of this checklist. Reference terms:

Agentic AI
AI systems where a model plans and takes actions through tools/APIs — not just answering, but doing. Security must constrain what the agent can do and prove what it did.
Amazon Vertex AI
Google Cloud managed service for foundation models and agents; provides model access, Agents, Knowledge Bases, and Guardrails.
Model Armor / Vertex AI safety filters
Configurable input/output filters for models: denied topics, PII/PHI redaction, and prompt-attack detection.
Vertex AI Agents
Managed agents that call tools (“action groups”) and knowledge bases under a scoped execution role.
Return of control
A Vertex AI pattern where the agent hands a proposed action back to your application (or a human) instead of executing it directly.
Prompt injection
An attack where untrusted content (a document, tool output, web page) smuggles instructions that hijack the agent. Treat all such content as untrusted.
RAG
Retrieval-Augmented Generation — grounding a model on retrieved documents (a vector store / knowledge base) rather than only its training.
MCP
Model Context Protocol — a standard for connecting agents to external tools/data servers; third-party MCP servers are supply-chain surface to vet.
Human-in-the-loop
A required human approval step before an agent performs a high-impact or regulated action.
Cloud IAM
Google Cloud Identity and Access Management — users, roles, and policies that govern who/what can call which Google Cloud APIs.
Cloud Identity + Workforce Identity Federation
Google Cloud SSO service; federates human access to your identity provider with permission sets instead of standing user-managed service-account keys.
Role / STS
An Cloud IAM identity assumed temporarily; STS issues short-lived credentials so nothing long-lived is stored.
Org Policy
Org Policy constraint — an organization-wide guardrail that caps what any account/principal can do, regardless of their Cloud IAM policy.
Cloud KMS / CMEK key
Key Management Service; a customer-managed key (CMEK key) encrypts data at rest with policies and rotation you control.
public access prevention + uniform bucket-level access
Account/bucket setting that prevents Cloud Storage data from being exposed publicly by policy or ACL.
VPC / VPC firewall rule
Virtual Private Cloud network isolation; a VPC firewall rule is a stateful virtual firewall on a resource’s network interface.
Private Service Connect / Private Service Connect
Private network paths to Google Cloud or partner services so traffic never traverses the public internet.
Cloud Armor / Cloud Armor DDoS defense
Google Cloud Web Application Firewall (managed rules, rate limiting) and Cloud Armor DDoS defense (DDoS protection) at the edge.
Security Command Center (Event Threat Detection)
Managed threat detection across accounts, workloads, and data planes (malware, GKE, Cloud SQL, Cloud Functions, Cloud Storage).
Security Command Center (posture)
Cloud security posture management — aggregates findings and scores config against benchmarks (CIS GCP Benchmark, CIS).
Config Validator + Org Policy
Records resource configuration and evaluates it against rules/posture constraints continuously.
Cloud Audit Logs
Records every Google Cloud API call — the primary audit log; can be made immutable in a locked log-archive account.
Sensitive Data Protection (Cloud DLP)
Managed sensitive-data discovery for PII/PHI in Cloud Storage.
Artifact Analysis / SCC vuln scanning
Continuous vulnerability scanning for EC2, containers, and Cloud Functions.
Secret Manager
Stores and rotates secrets; applications fetch them at runtime by role instead of embedding them.
a landing-zone blueprint (Assured Workloads)
Sets up and governs a multi-account landing zone with guardrails.
Assured Workloads / compliance reports
Automates evidence collection mapped to frameworks (SOC 2, HIPAA, etc.).
SOC 2 Type II
An attestation over a period that controls meeting the Trust Services Criteria (TSC) operate effectively. CC = Common Criteria; A/C/PI/P = Availability, Confidentiality, Processing Integrity, Privacy.
CMMC Level 1
The basic-safeguarding tier (17 practices from FAR 52.204-21) protecting Federal Contract Information (FCI). Physical (PE) practices are largely inherited from Google Cloud.
FCI
Federal Contract Information — non-public info provided/generated under a federal contract; the data CMMC L1 protects.
HIPAA Security / Privacy Rule
Security Rule = administrative (§164.308), physical (§164.310), technical (§164.312) safeguards for ePHI. Privacy Rule = use/disclosure limits, minimum necessary (§164.502), de-identification (§164.514).
BAA
Business Associate Agreement — required with Google Cloud for services processing PHI.
EU CRA
EU Cyber Resilience Act — essential cybersecurity requirements for products with digital elements. Annex I Part I = product requirements; Part II = vulnerability handling (incl. coordinated disclosure).
CMMC Level 2
The advanced tier — all 110 NIST SP 800-171 practices — required to handle Controlled Unclassified Information (CUI). Where CMMC L1 stops, L2 adds encryption, deep audit, configuration and incident-response practices; controls that ‘exceed L1’ here are covered at L2.
GDPR
EU General Data Protection Regulation. This checklist maps the technical security-of-processing measures (Art. 32) and related articles (5, 25, 30, 33/34, 44); GDPR also requires non-technical measures — lawful basis, consent, data-subject rights (Art. 15–22), DPO, DPIAs — that sit outside a technical checklist.
SBOM
Software bill of materials — a component inventory; a CRA expectation for products with digital elements.
Google Cloud Architecture Framework Security Pillar
Google Cloud’s security best-practice framework (SEC01–SEC11): identity, detection, infrastructure/data protection, incident response.

Design decisions — agentic-AI specific

  • The agent is a principal, not a person — it gets its own least-privilege role, its own audit identity, and never a human's credentials. (access control across every framework)
  • All model, tool, and RAG content is untrusted input — Guardrails + output validation defend against prompt injection and data exfiltration. (SOC 2 PI · CRA integrity · HIPAA)
  • High-impact actions require a human — destructive, financial, or PHI actions pass through an explicit approval step. (HIPAA §164.312(a) · CMMC AC.L1-3.1.2)
  • Every action is provable — full trace logging (prompt, retrieval, tool call, output) to an immutable store is the audit and assessor evidence. (HIPAA §164.312(b) · SOC 2 CC7.2)
  • Blast radius is bounded by design — per-tool sandboxes, per-tenant data isolation, egress allow-lists, and rate/cost limits contain a compromised agent. (CRA attack-surface · availability)
  • Data stays where it must — region-pinned models, per-classification KBs, and minimum-necessary retrieval keep PHI/FCI inside the boundary. (HIPAA privacy · CMMC · CRA data minimization)

How to read it: the frameworks converge on the same engineering primitives, so building to Google Cloud best practices satisfies most of each at once. CMMC L1 is a 17-practice floor for FCI; CMMC L2 adds the full 110 NIST 800-171 practices for CUI and picks up everything that “exceeds L1”. HIPAA adds privacy, minimum-necessary and a BAA. EU CRA adds product duties (no known exploitable vulns, SBOM, secure-by-default, coordinated disclosure). GDPR here covers security-of-processing (Art. 32) and related articles; data-subject rights, lawful basis and consent sit outside this technical checklist. SOC 2 II adds proof controls operated effectively over a period.
